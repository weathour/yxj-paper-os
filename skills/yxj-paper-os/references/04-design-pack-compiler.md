# Schema 0.3 Scoped Design-Pack Compiler Guide

This guide compiles the six-file detailed prewriting contract. It is not a workflow
engine, graph runtime, prose writer, automatic migrator, or semantic-quality judge.

## Authority Split

| Home | Authority |
|---|---|
| 00 Dimension Index | schema, scope identity, scope status, blockers, next action, output pointer |
| 00 Project Route | route constraints and ordinary/four-category decisions |
| 01 Materials Inventory | scientific M-* records and separate design-only TPL-* records |
| 02 Claim Evidence Boundary | sole scientific claim, support, uncertainty, and wording ceiling |
| 03 Writing Structure | authoritative section, paragraph, payload, important/frame, language, visual, edge, rule, budget, and scoped-plan records |
| 04 Writing Design Pack | compact counts, pointers, conditional template handling, prohibitions, and handoff |

04 never copies the authoritative 03 tables or a scope status.

## Detailed writer-ready Contract

A schema-0.3 scope is writer-ready only when:

- it owns one or more sections and every section owns ordered paragraphs;
- every paragraph has one function row and one payload/boundary row;
- every selected important paragraph has ordered controlled frames;
- all seven canonical detailed surfaces are satisfied or grounded not_applicable;
- referenced M-*, C-*, DEC-*, REQ-*, template, and detailed records resolve;
- triggered owner gates are resolved and routine grounded decisions are not
  misrepresented as owner gates;
- template sources remain behind the design_only firewall;
- no stale/candidate/blocked dependency affects the scope;
- blocker and next action are none and the output pointer resolves;
- no manuscript-ready prose, caption, title, abstract, or sentence is present.

Validation proves declared structure and traceability only.

## Pointer-only 04 Compilation

For each unique scope in 00:

1. emit exactly seven Detailed Surface Coverage rows;
2. emit exactly one Template Analysis Handling row;
3. emit exactly one Scoped Handoff row;
4. compute counts from 03 without copying authoritative records;
5. retain only applicable semantic, quantitative, or generic pointers;
6. keep 00 as the sole status authority.

Template Analysis Handling uses exactly semantic_primary,
semantic_plus_quantitative, quantitative_only, generic_fallback, or not_applicable.
not_applicable requires semantic, quantitative, and generic pointers to be none plus
a substantive rationale. Every mode keeps Firewall state design_only.

Scoped Handoff uses exactly:

Scope ID | 00 scope-row pointer | Detailed coverage pointer | Output pointer |
Active blocker IDs | Downstream prohibitions | Handoff note

It contains no copied status column. For writer-ready, Output pointer equals the 00
scope row output pointer and blockers are none.

When semantic template design is active, validate the hidden dossier read-only before
compilation and require exact 01 `TPL-*` and 03 semantic `TRULE-*` projection
agreement. Keep accepted annotations and optional analyzer artifacts separately typed.
Missing analyzer output is neutral for semantic-only, generic-fallback, or grounded
template-`not_applicable` handling.

## Schema 0.2 Recompilation

Schema 0.2 section-level readiness is not current detailed readiness. Migration is
agent-led and non-destructive; there is no automatic migration or placeholder-filling
command.

1. checkpoint or copy the workspace, or work on a version-controlled branch;
2. preserve valid route, material, claim, evidence, source, limitation, and existing
   structure records;
3. change the marker to schema 0.3 only while populating detailed surfaces;
4. set affected scopes to partial or blocked;
5. construct section/paragraph, payload, frame, language, visual, edge, rule, and
   budget records from current repository evidence;
6. perform template reading only when applicable and resolve only triggered owner
   gates;
7. restore writer-ready only after schema-0.3 validation passes.

No prior writer-ready declaration is carried forward automatically. Never rewrite,
delete, or silently normalize the source workspace while validating.

## Dependency Recheck

When a DEC-*, M-*, C-*, TPL-*, TRULE-*, or detailed record changes, record only
affected scopes/records and recompile their counts/pointers. Do not infer transitive
closure or globally stale unrelated scopes.

## Compiler Actions

Use inspect, derive, project, ask, and validate as unordered action families, never as
an exposed stage trace. Ask one focused question only for scientific_commitment,
argument_spine, material_local_tradeoff, or deliberate_divergence.

Do not draft manuscript prose, invent science/sources/results/visuals, execute
downstream skills, or claim research/manuscript/submission/publication/semantic
readiness.
